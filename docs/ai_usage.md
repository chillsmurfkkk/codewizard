# Використання LLM під час розробки проєкту

## Короткий опис

Під час розробки `codewizard` активно використовувалися LLM, зокрема Codex з GPT-5.5 (high reasoning).
LLM застосовувалася як інструмент для проєктування, написання коду, пояснення технічних рішень, 
пошуку помилок і підготовки документації

За моєю приблизною оцінкою, близько 90% коду проєкту було написано за допомогою Codex. При цьому
ідея застосунку, загальний напрям розробки, вибір функціональності, перевірка результатів і
фінальні рішення залишалися за мною

## Як саме використовувалася LLM

1. **Проєктування архітектури**
   - LLM подала ідею розбити застосунок на окремі модулі: сканування файлів, chunking, embeddings,
     vector store, retrieval, prompt building, LLM client і GUI
   - Через діалог уточнювалися межі MVP та наступні кроки розвитку

2. **Написання коду**
   - Модель допомагала реалізовувати класи, функції, обробку помилок, роботу з JSON, HTTP-запити,
     файлову систему та GUI-логіку
   - Код після генерації переглядався, запускався і коригувався

3. **Налагодження і виправлення помилок**
   - LLM допомагала аналізувати помилки компіляції та некоректну поведінку
   - Частина змін вносилася ітеративно: спочатку генерувалося рішення, потім воно перевірялося
     збіркою або запуском застосунку

4. **Покращення UX**
   - LLM використовувалася для покращення структури інтерфейсу, статусних повідомлень,
     debug-панелей, відображення джерел і prompt context
   - Окремо уточнювалися сценарії, коли кнопки мають бути вимкнені або коли користувачу треба
     показати зрозумілу помилку

5. **Документація**
   - Codex допомагав описувати архітектуру, поточний стан проєкту, RAG-пайплайн і обмеження; 
     був створений відповідний markdown файл docs/llm_project_context для "пам'яті" ШІ
     і використовувся як робочий контекст для подальшої розробки

## Роль розробника

LLM не замінила розробника повністю. Я сформулював вимоги, оцінював запропоновані
рішення, запускав збірку, перевіряв поведінку застосунку і приймав фінальні рішення щодо того,
що саме імплементувати

Фактично LLM виконувала роль асистента з програмування: швидко пропонувала варіанти реалізації,
допомагала тримати контекст проєкту і зменшувала кількість ручної роботи

## Обмеження такого підходу

- Згенерований код все одно потребує перевірки
- LLM може пропонувати рішення, які компілюються не з першої спроби або не повністю відповідають
  задуму
- Архітектурні рішення потрібно контролювати вручну, щоб проєкт не ставав надто складним,
  LLM люблять роздувати код, щоб покрити все "одним махом"
- Важливо перевіряти роботу програми збіркою, запуском і ручним тестуванням

## Висновок

У цьому проєкті LLM була одним з основних інструментів розробки. Вона суттєво прискорила написання
коду, документації та виправлення помилок

На мою думку, LLM інструменти досягли такого рівня розвитку, що можуть створювати робочі та безпечні
продукти з мінімальним супервайзингом зі сторони людини
